\textbf{\color{accentblue}The headline.}
We benchmarked a 19-million-row analytics workload on a 3-replica AWS
ClickHouse Cloud service. Each section below shows
\textbf{a schema}, \textbf{a query}, \textbf{the measured result},
and the lesson worth carrying into review next Monday.
The wins are concrete: $613\times$ faster inserts, $4.8\times$ faster
dashboards, $-37\%$ memory on small JOINs, $11\times$ tax for getting
mutations wrong.
Every number links to a JSON file in
\href{run:results/}{\texttt{results/}};
the supporting \emph{why} for each lesson is in the 15-lesson
\href{run:docs/}{\texttt{docs/}} folder.

\section{Batch your inserts}
% =========================================================================

\textbf{The setup.} A pipeline ingests a million customer events.
The wrong batch size won't error — it'll just stay a step behind real-time
forever, and one day pages oncall with \texttt{Too many parts}.

\schemaHd
\begin{lstlisting}
CREATE TABLE events (
    id UInt64, user_id UInt64,
    event_type LowCardinality(String),
    event_time DateTime
) ENGINE = MergeTree() ORDER BY (user_id, event_time);
\end{lstlisting}

\queryHd
We wrote 50{,}000 rows in four batch shapes — 100 / 1{,}000 / 10{,}000 /
50{,}000 rows per \texttt{INSERT} — and timed each.

\resultsHd

\begin{figure}[ht]
\centering
\begin{tikzpicture}
\begin{axis}[
    width=0.95\linewidth, height=4cm,
    xbar, bar width=14pt,
    enlarge y limits=0.25,
    xmin=0, xmax=215,
    xlabel={\small Throughput (thousands of rows / second)},
    symbolic y coords={b100, b1K, b10K, b50K},
    yticklabels={\texttt{batch=100}, \texttt{batch=1K}, \texttt{batch=10K}, \texttt{batch=50K}},
    ytick=data,
    nodes near coords,
    nodes near coords style={font=\footnotesize},
    nodes near coords align={horizontal},
    axis line style={rulegrey}, tick style={rulegrey},
    every axis label/.append style={font=\footnotesize, color=rulegrey},
    tick label style={font=\footnotesize}
]
\addplot[fill=accentgreen!75, draw=accentgreen] coordinates {
    (0.318,b100) (5.700,b1K) (63.772,b10K) (194.966,b50K)
};
\end{axis}
\end{tikzpicture}
\end{figure}

\vspace{-0.6em}
Insert throughput climbs from 318 rows/s at \texttt{batch=100} to
194{,}966 rows/s at \texttt{batch=50K}.
Source: Phase~A
(\href{run:results/benchmark_20260503_191546.json}{benchmark JSON}).

\lesson{%
\textbf{$613\times$ throughput from one parameter change.}
Every \texttt{INSERT} creates a part; small batches create small parts
faster than the merger can consume them.
Batch to 10{,}000--100{,}000 rows on the client, or set
\texttt{async\_insert=1} server-side and let ClickHouse buffer for you.
Single-row inserts in a loop are the fastest way to take a cluster down.}

\section{Pick the right \texttt{ORDER BY}}
% =========================================================================

\textbf{The setup.}
Three teams build the \emph{same} \texttt{events} table with three
different \texttt{ORDER BY} keys. Each team thinks theirs is fine — until
the analyst writes a query.

\schemaHd
\begin{lstlisting}
-- Three variants of the same table.
CREATE TABLE events_user_first (...) ENGINE = MergeTree
  ORDER BY (user_id, event_time);          -- team A
CREATE TABLE events_time_first (...) ENGINE = MergeTree
  ORDER BY (event_time, user_id);          -- team B
CREATE TABLE events_type_first (...) ENGINE = MergeTree
  ORDER BY (event_type, event_time, user_id); -- team C
\end{lstlisting}

\queryHd
We ran the three filter shapes the analyst actually writes:
\begin{lstlisting}
SELECT count() FROM <events> WHERE user_id = 42;
SELECT count() FROM <events> WHERE event_time >= now() - INTERVAL 7 DAY;
SELECT count() FROM <events> WHERE event_type = 'click';
\end{lstlisting}

\resultsHd
At 10M rows, each variant wins exactly the query that aligns with
its leading column:

\renewcommand{\arraystretch}{1.20}
\begin{tabularx}{\linewidth}{@{}X r r r r@{}}
\toprule
\textbf{ORDER BY}                               & \textbf{Storage} & \texttt{by\_user} & \texttt{by\_time} & \texttt{by\_type}\\
\midrule
\rowcolor{accentgreen!12}\texttt{(user\_id, event\_time)} & \textbf{287.7 MB} & \textbf{41.4 ms} & 64.5 & 60.5\\
\texttt{(event\_time, user\_id)}                          & 309.4 MB          & 43.1             & \textbf{40.9}    & 61.4\\
\texttt{(event\_type, event\_time, user\_id)}             & 308.5 MB          & 43.4             & 46.4             & \textbf{42.2}\\
\bottomrule
\end{tabularx}

\smallskip
The user-id-first variant is also \textbf{7\% smaller} on disk (granules
sort more cohesively under that key). Source: Phase~F
(\href{run:results/compare_features_20260503_194824.json}{ordering JSON}).

\lesson{%
\textbf{The first column of \texttt{ORDER BY} is the question you're
making fast.} Pick it to match your highest-frequency \texttt{WHERE}.
And pick once: \texttt{ORDER BY} is \emph{immutable} after table
creation — ``fixing it later'' is a full-data migration.}

\section{Use real types, not \texttt{String} for everything}
% =========================================================================

\textbf{The setup.}
A common shortcut at prototype time: declare every column \texttt{String}
and parse later. Cheap to type, expensive to scan.

\schemaHd
\begin{lstlisting}
-- Wrong: every column is a String
CREATE TABLE users_string (
    id String, age String, signup_date String, is_active String, ...
) ENGINE = MergeTree() ORDER BY id;

-- Right: native types match the data
CREATE TABLE users_native (
    id UInt64, age UInt8, signup_date Date, is_active UInt8,
    country LowCardinality(String), ...
) ENGINE = MergeTree() ORDER BY id;
\end{lstlisting}

\queryHd
Both tables hold the same 100{,}000 user rows from the same source.
We measured per-table compressed and uncompressed size, then ran
\texttt{SELECT count() WHERE country = 'US'} against each.

\resultsHd
\renewcommand{\arraystretch}{1.20}
\begin{tabularx}{\linewidth}{@{}X r r r@{}}
\toprule
\textbf{Schema} & \textbf{Compressed} & \textbf{Uncompressed} & \textbf{Filter latency}\\
\midrule
\texttt{String}-for-everything                          & 3.56 MiB & 14.20 MiB & 46.2 ms\\
\rowcolor{accentgreen!12}Native (\texttt{UInt64}, \texttt{Date}, \texttt{LowCardinality}, \ldots)
                                                        & \textbf{2.94 MiB} & \textbf{9.73 MiB} & 44.4 ms\\
\bottomrule
\end{tabularx}

\smallskip
Source: Phase~C7
(\href{run:results/experiment_c7_string_vs_native_20260503_192524.json}{c7 JSON}).

\lesson{%
\textbf{Native types use $32\%$ less raw I/O.} Free win, no code change
needed elsewhere — just declare \texttt{UInt64} instead of \texttt{String}
for IDs, \texttt{Date} for dates, \texttt{LowCardinality(String)} for
enums (countries, statuses, event types). Compressed sizes converge
because LZ4 papers over the difference, but the bytes flowing through
RAM and CPU don't.}

\section{One materialised view per dashboard}
% =========================================================================

\textbf{The setup.}
A dashboard refreshes ``events per day, broken down by type'' every minute.
Without help, it scans the whole 10M-row events table on every load.

\schemaHd
\begin{lstlisting}
-- 1. Tiny target table that holds aggregated state
CREATE TABLE events_hourly (
    day Date, event_type LowCardinality(String),
    cnt AggregateFunction(count, UInt64)
) ENGINE = AggregatingMergeTree() ORDER BY (day, event_type);

-- 2. Trigger that runs the GROUP BY at insert time
CREATE MATERIALIZED VIEW mv_events_hourly TO events_hourly AS
SELECT toDate(event_time) AS day, event_type,
       countState(toUInt64(id)) AS cnt
FROM events GROUP BY day, event_type;
\end{lstlisting}

\queryHd
\begin{lstlisting}
-- BEFORE: full scan over 10M rows
SELECT toDate(event_time), event_type, count()
FROM events GROUP BY 1, 2;

-- AFTER: read pre-aggregated rows from the MV target
SELECT day, event_type, countMerge(cnt)
FROM events_hourly GROUP BY day, event_type;
\end{lstlisting}

\resultsHd
\renewcommand{\arraystretch}{1.20}
\begin{tabularx}{\linewidth}{@{}X r r@{}}
\toprule
\textbf{Read path}                                    & \textbf{Latency} & \textbf{Storage}\\
\midrule
Full scan over 10M raw events                         & 196.7 ms & 74.3 MiB\\
\rowcolor{accentgreen!12}MV target (pre-aggregated)   & \textbf{41.0 ms} (\textbf{4.8$\times$ faster}) & $<$1 KiB\\
\bottomrule
\end{tabularx}

\smallskip
The MV runs the \texttt{GROUP BY} once per insert; reads are
already-aggregated rows. Source: Phase~F
(\href{run:results/compare_features_20260503_194828.json}{materialized\_views JSON}).

\lesson{%
\textbf{Build one MV per dashboard query.} Dashboard reads jump from
seconds to milliseconds, regardless of how big the source table grows.
The cost is per-insert: every new row runs the MV's \texttt{GROUP BY}
once. It's the best deal in ClickHouse.}

\section{Replace small JOINs with a dictionary}
% =========================================================================

\textbf{The setup.}
A daily report joins 300{,}000 orders against a 100{,}000-row users
table to break revenue down by country. The JOIN runs every morning,
gets slower every quarter as users grow.

\schemaHd
\begin{lstlisting}
-- The dictionary: in-memory, refreshed every 5 minutes
CREATE DICTIONARY users_dict (
    id UInt64, country String, segment String
)
PRIMARY KEY id
SOURCE(CLICKHOUSE(TABLE 'users'))
LAYOUT(HASHED())
LIFETIME(MIN 300 MAX 360);
\end{lstlisting}

\queryHd
\begin{lstlisting}
-- BEFORE: classic JOIN
SELECT u.country, count(), sum(o.total_price)
FROM orders o INNER JOIN users u ON u.id = o.user_id
GROUP BY u.country;

-- AFTER: same answer via dictGet
SELECT dictGet('users_dict', 'country', user_id) AS country,
       count(), sum(total_price)
FROM orders
GROUP BY country;
\end{lstlisting}

\resultsHd
\renewcommand{\arraystretch}{1.20}
\begin{tabularx}{\linewidth}{@{}X r r r r@{}}
\toprule
                            & \textbf{Wall} & \textbf{Server} & \textbf{Rows read} & \textbf{Memory}\\
\midrule
\texttt{INNER JOIN}         & 56.8 ms & 19 ms & 400{,}000 & 27.8 MB\\
\rowcolor{accentgreen!12}\texttt{dictGet}
                            & \textbf{49.7 ms} & \textbf{11 ms} & \textbf{300{,}000} & \textbf{17.4 MB}\\
\textit{Improvement}        & 1.14$\times$ & 2$\times$ & $-25\%$ & $-37\%$\\
\bottomrule
\end{tabularx}

\smallskip
Source: Phase~C1
(\href{run:results/experiment_c1_dictionary_vs_join_20260503_192050.json}{c1 JSON}).

\lesson{%
\textbf{$-37\%$ memory and twice as fast server-side, same answer.}
\texttt{dictGet} skips the hash-table build a JOIN pays every time;
the dictionary is loaded once and reused across queries. The pattern
fits any small lookup table that changes slowly: countries, segments,
status maps, geography.}

\section{Turn on the query cache for repeated dashboards}
% =========================================================================

\textbf{The setup.}
Five users hit the same heavy aggregation in five tabs at 9:01~am
every Monday. Without help, the cluster recomputes the same answer
five times.

\schemaHd
The cache is a setting, not a schema change. Default off; flip to on
per query, per session, or per user.

\queryHd
\begin{lstlisting}
SELECT toStartOfHour(timestamp) AS hour, metric_name,
       avg(value), quantile(0.95)(value)
FROM metrics
GROUP BY hour, metric_name
ORDER BY hour, metric_name LIMIT 1000
SETTINGS use_query_cache = 1, query_cache_ttl = 60;
\end{lstlisting}

\queryHd
\begin{lstlisting}
-- Confirm it worked
SELECT query_cache_usage, query_duration_ms
FROM system.query_log
WHERE query_id IN (...) ORDER BY event_time DESC;
-- Returns 'Write' on the first run, 'Read' on the rest.
\end{lstlisting}

\resultsHd
\renewcommand{\arraystretch}{1.20}
\begin{tabularx}{\linewidth}{@{}X r r l@{}}
\toprule
\textbf{Run}     & \textbf{Wall} & \textbf{Server} & \textbf{Cache}\\
\midrule
1 (cold, miss)   & 68.6 ms       & 28 ms           & \texttt{Write}\\
\rowcolor{accentgreen!12}2--5 (avg, hits) & \textbf{41.8 ms} & \textbf{1 ms} (\textbf{28$\times$}) & \texttt{Read}\\
\bottomrule
\end{tabularx}

\smallskip
\texttt{system.events} confirms 1~miss and 4~hits. Source: Phase~C2
(\href{run:results/experiment_c2_query_cache_20260503_192130.json}{\texttt{c2 JSON}}).

\lesson{%
\textbf{One setting, $28\times$ server-side speedup on hits.}
The default 60-second TTL is the right shape for dashboards.
Don't enable it for queries with \texttt{now()}, \texttt{rand()},
\texttt{dictGet}, or anything reading \texttt{system.*} — those
results are non-deterministic and skip the cache anyway.}

\section{Don't mutate. Insert new versions.}
% =========================================================================

\textbf{The setup.}
A 100{,}000-row \texttt{users} table needs 1\% of rows updated to
\texttt{status = 'inactive'}. Two equivalent designs; one of them
will rewrite an entire data part on disk.

\schemaHd
\begin{lstlisting}
-- Approach A: plain MergeTree, mutate in place
CREATE TABLE users_mut (
    id UInt64, name String, status LowCardinality(String)
) ENGINE = MergeTree() ORDER BY id;

-- Approach B: ReplacingMergeTree, append new versions
CREATE TABLE users_rmt (
    id UInt64, name String, status LowCardinality(String),
    updated_at DateTime DEFAULT now()
) ENGINE = ReplacingMergeTree(updated_at) ORDER BY id;
\end{lstlisting}

\queryHd
\begin{lstlisting}
-- Approach A: rewrites the whole 258 KiB part
ALTER TABLE users_mut UPDATE status = 'inactive' WHERE id < 1000;

-- Approach B: appends a 4 KiB part with the new versions
INSERT INTO users_rmt (id, name, status, updated_at)
SELECT id, name, 'inactive', now() FROM users_rmt WHERE id < 1000;

-- Read the latest version (no FINAL needed, faster than FINAL):
SELECT id, argMax(status, updated_at) AS status
FROM users_rmt GROUP BY id;
\end{lstlisting}

\resultsHd
\renewcommand{\arraystretch}{1.20}
\begin{tabularx}{\linewidth}{@{}X r r r@{}}
\toprule
\textbf{Approach}                       & \textbf{Wall} & \textbf{Parts changed} & \textbf{Bytes touched}\\
\midrule
\texttt{ALTER TABLE \ldots UPDATE}      & 1{,}060 ms    & 1 (rewritten)         & 258 KiB\\
\rowcolor{accentgreen!12}\texttt{INSERT} into \texttt{ReplacingMergeTree}
                                        & \textbf{96 ms} (\textbf{11$\times$ faster}) & 1 (appended) & 4 KiB\\
\bottomrule
\end{tabularx}

\smallskip
Source: Phase~C5
(\href{run:results/experiment_c5_mutation_cost_20260503_192410.json}{c5 JSON}).

\lesson{%
\textbf{Mutations rewrite parts; \texttt{ReplacingMergeTree} appends.}
$11\times$ less work for the same logical change at this scale, and
the gap widens with table size. Read with \texttt{argMax(col, version)
GROUP BY pk} to get the latest version cleanly. Save \texttt{ALTER UPDATE}
for one-off corrections, never for steady-state writes.}

\section*{Bonus: how to crash your cluster in one INSERT}
% =========================================================================
\addcontentsline{toc}{section}{Bonus: how to crash your cluster in one INSERT}

\textbf{The setup.}
The events spread over a multi-year window (Faker timestamps).
Someone partitions by day, thinking it'll speed up time-range queries.
The first \texttt{INSERT} catches fire.

\schemaHd
\begin{lstlisting}
-- Looks reasonable. Isn't.
CREATE TABLE events_part_day (...)
ENGINE = MergeTree()
PARTITION BY toDate(event_time)        -- one partition per day
ORDER BY (user_id, event_time);
\end{lstlisting}

\queryHd
\begin{lstlisting}
INSERT INTO events_part_day SELECT * FROM events;
\end{lstlisting}

\resultsHd
The server lectures us with its own canonical error message before a
single byte reaches disk:

\begin{tcolorbox}[colback=accentred!8, colframe=accentred,
                  boxrule=0pt, leftrule=3pt, arc=2pt,
                  left=10pt, right=10pt, top=4pt, bottom=4pt,
                  enhanced jigsaw, fontupper=\small]
\textit{Code: 252. DB::Exception: Too many partitions for single INSERT
block (more than 100). \ldots Large number of partitions is a common
misconception. It will lead to severe negative performance impact,
including slow server startup, slow INSERT queries and slow SELECT
queries. \ldots Please note, that partitioning is not intended to speed
up SELECT queries (\texttt{ORDER BY} key is sufficient \ldots).
Partitions are intended for data manipulation
(\texttt{DROP PARTITION}, etc).}
\end{tcolorbox}

\smallskip
Captured verbatim in
\href{run:results/compare_features_20260503_191740.json}{\texttt{partitioning JSON}}.

\lesson{%
\textbf{Partitioning is for \texttt{DROP PARTITION}, not for query
speed.} \texttt{ORDER BY} steers query latency; partitions steer the
\emph{lifecycle} of your data (retention, archive, tiered storage).
Aim for 100--1{,}000 distinct partitions total --- monthly
(\texttt{toYYYYMM}) covers most cases. The day you partition by
\texttt{user\_id} or by raw date is the day Cloud pages you.}

\section*{The five-step playbook}
% =========================================================================
\addcontentsline{toc}{section}{The five-step playbook}

The lessons above compose into a single review checklist. Apply in
order; each step builds on the one before it.

\medskip
\renewcommand{\arraystretch}{1.20}
\begin{tabularx}{\linewidth}{@{}c X@{}}
\toprule
\textbf{\#} & \textbf{Step}\\
\midrule
1 & \textbf{Plan \texttt{ORDER BY} before \texttt{CREATE TABLE}.}
    Low-cardinality columns first, high-cardinality last.
    The leading column is the question you're optimising for.\\
2 & \textbf{Use native types and \texttt{LowCardinality} for enums.}
    Avoid \texttt{Nullable} unless the absence is semantic.\\
3 & \textbf{Batch inserts at 10K--100K rows} (or set
    \texttt{async\_insert=1, wait\_for\_async\_insert=1}).
    Never insert single rows in a loop.\\
4 & \textbf{Build an aggregating MV per dashboard query.}
    Use \texttt{*State}/\texttt{*Merge} pairs in the MV definition.\\
5 & \textbf{Reach for \texttt{dictGet}, query cache, and skip indexes
    only after} \texttt{EXPLAIN ESTIMATE} \textbf{shows the primary
    index alone isn't enough.}\\
\bottomrule
\end{tabularx}

\medskip
And three things to flag in PR review:
\begin{itemize}
\setlength\itemsep{0pt}
\item \texttt{ALTER}-style \texttt{UPDATE} or \texttt{DELETE} for routine
      writes --- use \texttt{ReplacingMergeTree} or
      \texttt{DROP PARTITION} instead.
\item \texttt{OPTIMIZE TABLE \ldots FINAL} on a recurring schedule ---
      background merges already do that work.
\item High-cardinality partition keys (\texttt{toDate} on multi-year
      data, \texttt{user\_id}) --- aim for 100--1{,}000 partitions
      total.
\end{itemize}
